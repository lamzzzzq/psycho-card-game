# 交互沙盒布局修复 · 交付需求文档

> 目标读者：接手改这个的开发者（可无本项目上下文）。
> 一句话需求：**让交互式教学沙盒的每一步都在「同一个视窗内完整显示、手牌位置固定不跳、任何一步都不被截断」。**

---

## 1. 背景 / 为什么要改

- 组件：`人格麻將` 教学页里的**交互式沙盒**（一个带金色高亮引导的分步教程）。
- 用途：① 学生自学游戏流程；② 我们要拿它录一个 1 分钟的走查视频。
- 现状缺陷：**不同教学步骤的 UI 高度差异很大**，导致页面在步骤间「忽高忽低」，其中**「自摸碰」这一步的手牌第二行会被挤出视窗、被底部固定指引栏截断**。录屏时页面跳动 + 手牌看不全，无法使用；学生实际操作到这一步时也看不全手牌。

**症状示例**（同一沙盒不同步骤，视窗高度不变）：
- 「查看 2 张」步：操作区只有一行按钮 → 内容矮，手牌完整。
- **「自摸碰」步：操作区 = 大标题横幅 +「第一步·选维度」面板 + 目标板 → 内容超高，手牌第二行溢出视窗、被指引栏盖住。** ← 主要 bug
- 「截胡碰」步：操作区 = 横幅 + 大卡片提示 → 中高。
- 「食胡」步：操作区 = 横幅 + 目标板 → 矮，居中。

- 状态机 scene 类型定义：约 `line 77`（`type Scene = 'start' | 'viewing' | ...`）。

  - `line 444` `<motion.div className="psy-panel ... p-5 pb-8 ... mb-44">` — **沙盒主面板**（`mb-44` = 硬编码底部留白给指引栏）。
  - `line 473` `<div className="space-y-4 ... p-4">` — **牌桌纵向堆叠容器**（高度随内容变）。
    - `line 475` 归档区 chips
    - `line 517` 操作大标题横幅（自摸碰/截胡碰/食胡）— **仅部分 scene 出现**

    - `line 706` **手牌区** — 挂在这个可变高度堆叠的最底部

## 3. 根本原因

```
主面板 (mb-44 硬留白)
└─ 牌桌 (space-y-4, 高度 = 内容总高, 无上限)
   ├─ 归档区
   ├─ 操作横幅        ← 高度按 scene 变（0 ~ 一大条）
   ├─ 操作面板        ← 高度按 scene 变（一行按钮 ~ 选维度面板/大卡片）  ← 元凶
   ├─ 目标板
   └─ 手牌            ← 挂在最底部, Y 位置随上方总高浮动
指引栏 (fixed bottom-0)  ← 盖在视窗底部
```

两个问题叠加：
1. **手牌的垂直位置不固定**：它在一个「高度随内容增长」的纵向流里，操作区一高，手牌就被往下推 → 每步位置都在跳。
2. **没有视窗高度约束**：牌桌总高可以超过视窗；一旦超过，底部（手牌）就滑到 `fixed` 指引栏后面被**截断**。`mb-44` 是按固定指引栏高度猜的硬编码留白，扛不住高操作面板。

---

## 4. 要求的修复

**核心思路：给操作区一个「预留固定高度的槽位」，把手牌锚定在固定位置，整个沙盒锁进一个视窗高度内。**

### 方案 A（推荐，改动小）：操作区预留等高槽位

1. 把**操作横幅 + 操作面板**这一段（约 `line 517~700`）包进一个**固定 `min-height` 的容器**，高度设为「最高那一步（自摸碰选维度）所需的高度」。
   - 矮的步骤（如「查看 2 张」）渲染进同一槽位，多出来的是空白 → 目标板和手牌的 Y 位置**在所有步骤间保持一致**。
   - 建议该槽位内容**垂直居中**，视觉更稳。
2. 目标板 + 手牌放在槽位下方，位置因此固定不动。
3. 用真正的布局给指引栏留空间，**替换掉 `mb-44` 硬编码**（例如：底部留白 = 指引栏实测高度，或把指引栏高度设为 CSS 变量统一引用）。

> 优点：改动局部、风险低。缺点：`min-height` 值要按最高步骤实测确定，未来新增更高的步骤要复核。

### 方案 B（更稳，改动大）：视窗锁定的 flex 框架

1. 沙盒主面板改成 `height: 100dvh`（或 `max-height`）+ `flex flex-col`。
2. 顶部（标题/归档）+ **操作区 `flex-1 min-h-0 overflow-y-auto`**（万一某步超高，只在操作区内部滚动，不影响其它区）。
3. **手牌 + 目标板做成 sticky/固定的底部 footer**，永远压在指引栏正上方、始终完整可见。

> 优点：对任意高度的步骤都健壮、手牌永远不被截断。缺点：改动大，要重排整个沙盒的三段式结构。

**推荐先做方案 A**（能解决当前所有已知问题、成本低）；若后续步骤高度差异继续扩大，再升级到方案 B。

## 5. 各步骤操作区高度参考（定 `min-height` 用）

| Scene | 操作区内容 | 高度 |
|---|---|---|
| `start` / `viewing` / `view-picking` / `after-draw` / `pong-success` / `discard-confirm` / `turn-done` | 一行按钮 | 矮 |
| **`pong-dimension`** | 大横幅 +「第一步·选维度」面板（5 维 chip + 取消） | **最高（按它定 min-height）** |
| `pong-picking` | 大横幅 + 选牌提示 | 高 |
| `claim-window` | 大横幅 + 截胡大卡片提示 | 高 |
| `hu-window` | 大横幅 + 已归档 chips + 食胡卡 | 高 |

> `pong-dimension` 一般是最高的一步，`min-height` 以它为准。

## 6. 验收标准（可测）

在 **视窗高度 720 / 800 / 900 px** 三档下，逐一走完全部步骤，均需满足：

- [ ] **每一步都在一个视窗内完整显示**，无需滚动。
- [ ] **手牌全部可见**（含第二行），不被底部指引栏遮挡/截断——尤其「自摸碰」步。
- [ ] **手牌与目标板的垂直位置在所有步骤间保持一致**（步骤切换时不上下跳）。
- [ ] 底部指引栏始终可见、不与手牌重叠。
- [ ] 中文（繁体）与英文（`/?lang=en`）两种语言下都成立（文案变长不破坏布局）。
- [ ] 手机竖屏（约 390×844）下也不溢出/不截断（沙盒本就手机优先）。

---

## 7. 约束（不要动的东西）

- **不要改游戏/教学逻辑**：reducer、scene 状态机、各步的引导文案和高亮逻辑保持不变，**只调布局/尺寸/容器**。
- **保留金色高亮引导**（spotlight）和**底部固定指引栏**——它们是教学核心，只是别让它们截断手牌。
- **保留 i18n**：所有文案走 `TUTORIAL_T[loc]`，勿硬编码中文；繁/英都要过。
- **卡面插画不动**（`/cards/*.webp`）。

## 8. 自测方式

```bash
# 本地起开发服务器
npm run dev
# 浏览器打开教学页 → 进「交互沙盒」，逐步点完 7 步，
# 用开发者工具把视窗高度依次设 720 / 800 / 900，核对第 6 节验收项。
# 英文版：先访问 /?lang=en 再进 tutorial。
```

真实流程 7 步（照此逐步验证）：看目标张数板 → 抽牌 → 查看 2 张 → 自摸碰(选维度→选 4 张→确认) → 弃牌 → 截胡碰 → 食胡。
