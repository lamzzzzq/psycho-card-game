# 「輪次」定義與回合公平性 — 設計分析

> 一次設計討論的整理：如何計算「一輪」（輪次）、為什麼碰牌（截胡偷回合）與罰停跳過
> 在輪次計數上表現不一致、以及為什麼兩種輪次定義都無法保證每位玩家真的輪得到出牌。
> 內含目前程式碼的實作現狀。

---

## 1. 問題陳述

實測中浮現兩個相關的疑慮：

1. **輪次計數** —— 有碰牌（偷回合）時，「一輪」的意義變得模糊。棄牌被碰走時，指針
   跳到碰牌者，中間一或多個座位被略過。這個略過算不算跨過了一輪的邊界？
2. **後期死局** —— 玩家可能只剩一個維度未歸檔、手上卻一張該維度的牌都沒有，同時對手
   一直在歸檔。若碰牌不斷偷走回合，該玩家很少（甚至完全沒有）機會輪到抽牌，體驗很差。

這其實是**兩個不同的問題**，很容易被混為一談。

---

## 2. 目前「一輪」如何定義

`src/lib/game-logic.ts` 的 `advancePlayer()`：

```
nextPlayerIndex = (currentIndex + 1) % playerCount
isRoundEnd      = (nextPlayerIndex === 0)      // 座位 0
nextRound       = isRoundEnd ? currentRound + 1 : currentRound
```

**一輪 ＝ 出牌指針「正常順序」轉回座位 0。** 輪次計數器只在*正常推進*落到座位 0 時 +1。

### 誰是座位 0（基準）？

| 模式 | 座位 0 ＝ | 依據 |
|------|-----------|------|
| **聯機 PvP** | **房主** | `room-api.ts` 建房時把房主以 `seat_index: 0` 寫入；玩家依 `seat_index` 排序。 |
| **單機** | **人類玩家（你）** | 人類固定在 index 0（`currentPlayerIndex === 0` 即人類回合）。 |

所以輪次邊界被錨定在**某一個特定座位**上 —— 聯機是房主、單機是你。這正是下述不一致的根源。

## 3. 現狀：程式目前如何實現

### 3.1 罰停相關欄位（`Player`，見 `src/types/index.ts`）

| 欄位 | 意義 |
|------|------|
| `skipNextTurn` | 下一次輪到此座位時自動跳過一次。 |
| `frozenUntilOwnDiscard` | 凍結：跳過期間不能碰/胡/出牌，直到自己真正出過一次牌才解除。 |
| `extraSkipQueued` | 加重罰停旗標：被跳一次後重新激活 `skipNextTurn`（即再跳第二次）。 |

> 罰則（碰失敗 / 自摸碰失敗 / 食胡失敗）＝ 罰停 **2 個**自己的出牌位（透過 `extraSkipQueued`
> 實現「跳兩次」），跳完即解凍。

### 3.2 正常推進 —— `advancePlayer()`
逐格 +1；`nextPlayerIndex === 0` 時 `round + 1`。`totalRounds === 0` 表示無限。
`isGameOver` 在「輪次結束且 nextRound > totalRounds」時為真。

### 3.3 罰停跳過 —— `skipPenalizedPlayers()` ✅ 輪次一致

當指針落在被罰停 / 已退出的座位時，**逐格地走**，對每個被跳座位呼叫 `advancePlayer()`：

```
迴圈（上限 players.length * 3，容納加重罰停的第二跳）:
    若當前座位沒有 skipNextTurn 且沒有 hasLeft → 停止
    產生一條 'skip' 日誌（clearedPenalty 標記是否為最後一跳解凍）
    consumeOnePenaltySkip(該座位)          // 見下
    advancePlayer(...)                      // 走一格；經過座位 0 → round + 1
```

`consumeOnePenaltySkip()`：
```
if (extraSkipQueued)  → skipNextTurn = true, extraSkipQueued = false   // 排入第二跳
else                  → skipNextTurn = false, frozenUntilOwnDiscard = false  // 最後一跳、解凍
```

因為它**一格一格地走** `advancePlayer`，跳過期間**經過座位 0 就會讓 round +1**。
這忠實實現了原本的原則：

> **「被跳過，仍然算指針經過了你。」**

### 3.4 碰牌偷回合 —— `pongCard()`（「Option B」）❌ 輪次不一致

棄牌被碰走時，指針**直接跳**到碰牌者：

```
currentPlayerIndex = pongerIndex   // 直接跳，不逐格走
currentRound       = 不變           // ← 偷回合這條路完全沒動 round
consumeBypassedPenaltySkips(...)   // 只補處理「被略過且處於罰停中」的座位
```

`consumeBypassedPenaltySkips()`：
```
for i = discarder+1 … ponger（不含 ponger）:
    if (!players[i].skipNextTurn) continue;   // 只處理罰停中的座位
    consumeOnePenaltySkip(players[i])
    push 一條 'skip' 日誌（round 為當前值，未遞增）
```

因此，對於**一般（非罰停）**的被略過座位：
- **不會**產生 skip 日誌；
- 輪次計數器**不會**遞增，即使這次跳躍在物理上越過了座位 0。

> 邊角情況：若碰牌後碰牌者手牌歸零（最後兩張剛好都被碰掉），為避免死鎖，這條路會改走
> `advancePlayer` 前進到下一位 —— 該情況下 round **可能**遞增。碰牌直接食胡則進入 game-over。

所以碰牌這條路，對輪次計數器**違反**了「被跳過＝經過」的原則。

### 3.5 兩條路對照

| | 罰停跳過（Path A） | 碰牌偷回合（Path B） |
|---|---|---|
| 指針如何移動 | 逐格走 `advancePlayer` | 直接跳到碰牌者 |
| 經過座位 0 是否 round+1 | **是** | **否**（除非碰後手牌歸零那條邊角路） |
| 被略過座位是否記日誌 | 是（每個被跳座位） | 只有罰停中的座位；一般座位**無** |
| 是否符合「跳過＝經過」 | ✅ | ❌ |

## 4. 逐案分析

座位：**A(0，房主) → B(1) → C(2) → D(3)**，`totalRounds = 5`。

### 案例 1 — 正常循環（無碰、無罰停）
A→B→C→D→A…… 每次回到 A 就 round+1，每位玩家每輪各出一次。這是設計預期的正常基準。

### 案例 2 — 罰停跳過（Path A）
設 C 被罰停。指針到 C，逐格 `advancePlayer` 自動跳過；若跳過期間走過座位 0，round 遞增，
且 C 的跳過有日誌。**輪次計數正確，這次跳過「經過」了 C。** ✅

### 案例 3 — 碰牌略過（Path B）—— 你提出的場景
B 出 → C 出 → D 出。**D 出牌時被 B 碰走**：
- 指針 `D(3) → B(1)`，**座位 A(0) 被略過**。
- `currentRound` **不變** → **仍是本輪**。
- A **沒有** skip 日誌，本圈也沒輪到。

**「算下一輪還是本輪？」→ 本輪。** 而且 A 被靜默略過。這就是與案例 2 的不一致。

### 案例 4 — 某玩家全程被略過（公平性陷阱）
設 A、B、D 一直互相碰對方的棄牌，使指針永遠落不到 **C**，且 C 從無可碰的牌（永遠不是碰牌者）：

- C **從頭到尾沒輪到** —— 不曾抽牌、不曾出牌。
- 輪次計數器仍隨 A 的正常回合 / 越過座位 0 而遞增。
- **遊戲在第 5 輪結束，C 一張牌都沒出過。**

**這在目前程式碼裡就會發生**，與任何修法無關，因為輪次錨定在座位 0、**不追蹤 C 是否曾行動**。

## 5. 兩個獨立問題

| # | 問題 | 性質 |
|---|------|------|
| **P1** | 碰牌偷回合不把被略過座位計入輪次（與罰停跳過不一致）。 | 輪次計數 bug。 |
| **P2** | 玩家可能整局被略過、從未行動。 | 回合公平性缺陷。 |

**關鍵認知：** 輪次定義只是**遊戲結束的計時器**，它本身無法保證「誰輪得到出牌」的公平。
P1 與 P2 必須分開處理。

---

## 6. 修法選項

### 6.1 輪次定義（處理 P1）

**選項 1 — 維持座位 0 錨點，讓碰牌變一致。**
碰牌偷回合時，計算 `discarder → ponger` 的跳躍越過座位 0 幾次，據此遞增 `currentRound`；
並為被略過座位補上 skip 日誌。改動小而局部、維持現有語義。
- ⚠️ **不**解決 P2。案例 4 依然成立：玩家仍可能整局被略過、遊戲照樣按輪數結束。

**選項 2 — 把「一輪」重定義為「每位玩家各行動過一次」。**
只有當每個座位都至少行動過一次後才 round+1。更直觀：「5 輪」≈「每人各出約 5 次」。
- ⚠️ 同樣**不**解決 P2。若 C 一直被略過，這一輪**永遠湊不齊**，計數器卡住、遊戲**拖長**
  （直到有人胡或牌堆抽空），C 仍然沒出過牌。只是把失敗模式從「很快結束、C 沒玩到」
  換成「拖很久、C 還是沒玩到」。

**兩種輪次定義都不解決 P2。**

### 6.2 回合公平保障（處理 P2 —— 真正的解法）

與輪次計數正交。可選：

1. **連續跳過上限** —— 某座位被碰牌連續略過 N 次後，下一次碰牌**不得**再略過它；
   指針落到它身上讓它抽牌。（推薦 —— 直接杜絕「永遠沒玩到」。）
2. **越過補償** —— 被碰牌略過的座位，下次輪到時給一次保底抽牌。
3. **限制連碰** —— 限制同一條棄牌鏈上回合被偷走的次數。

---

## 7. 建議

當作兩個獨立改動：

- **輪次定義：** 傾向**選項 2**（依每人出牌次數計），讓「N 輪」對每位玩家意義一致，
  而非錨定在房主座位。（若只想消除 P1 不一致、不改語義，選項 1 是較省的路。）
- **公平性：** 加**連續跳過上限**（§6.2.1）。這 —— 而非輪次定義 —— 才是真正保證每位
  玩家輪得到出牌的機制。

任何改動都應附涵蓋案例 2–4 的單元測試。

## 8. 既有兜底（已在程式中）

- **輪數上限**（5/10/15）：限制遊戲長度；無人胡時按已歸檔維度數排名。（課堂建議避免 ∞。）
- **牌堆循環補充（不會流局）：** 抽牌堆抽空時，程式會把**棄牌堆洗回抽牌堆**（Fisher-Yates）
  重新可抽，`discardPile` 清空（`game-logic.ts:420-434`）。因此**正常情況下牌不會抽完、
  沒有「流局」**。唯一的牌數硬停是**抽牌堆與棄牌堆同時皆空**的極端情況（幾乎只在所有牌都
  在玩家手上/已歸檔時發生），此時 force game-over 以防無限迴圈。
- **牌庫配平：** 牌堆依人數縮放，使每維度張數足夠 —— 且因為棄牌會循環洗回，需要的牌其實
  一直在牌池裡流動、**不會消失**。但這只保證牌「存在且不斷回流」，不保證被牌荒卡住的玩家
  有「回合」去取得（那是 P2）。

> **對 P2 的放大效應：** 因為牌堆會循環補充、不會自然抽空，若設 ∞ 輪、又無人能胡、且某玩家
> 被永久略過，遊戲**理論上可以永遠打下去**。也就是說「牌堆抽空」這個兜底其實**不存在**；
> 真正能終止遊戲的只有「有人食胡」或「打滿有限輪數」。這讓 §6.2 的回合公平保障更重要。
